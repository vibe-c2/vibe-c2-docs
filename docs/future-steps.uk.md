# Майбутні кроки та ідеї

Ця сторінка відстежує заплановані подальші дії та відкриті ідеї, які ще не оформлені як ADR.

## Найближчі наступні кроки

- Визначити та затвердити базовий конверт повідомлення:
  - `message_id`
  - `correlation_id`
  - `type`
  - `version`
  - `timestamp`
  - `payload`
- Визначити топологію RabbitMQ:
  - стратегія exchange
  - routing keys
  - конвенції найменування черг
  - dead-letter та retry черги
- Зафіксувати межі володіння модулів:
  - що належить `channel`
  - що належить `implant provider`
  - що належить `translator`

## Ідеї для реалізації

- Створити пакет `contracts` зі спільними схемами повідомлень.
- Додати контрактні тести між основним сервером та модулями.
- Додати compose-профіль для кожного середовища (`dev`, `staging`, `lab`).
- Додати дашборд здоров'я модулів (затримка черги, частота помилок, латентність обробки).
- Додати матрицю сумісності версій модулів (ядро проти модулів).

## Ідеї з безпеки та надійності

- Забезпечити окремі облікові дані RabbitMQ для кожного модуля та дозволи за принципом найменших привілеїв.
- Впровадити підписування/перевірку цілісності повідомлень для чутливих потоків.
- Додати ключі ідемпотентності для шляхів виконання команд.
- Визначити стратегію захисту від повторного відтворення для затриманих/повторно надісланих повідомлень.

## Ідеї щодо робочого процесу документації

- Переводити стабільні ідеї з цієї сторінки в ADR.
- Зберігати цю сторінку як проміжну область для архітектурних обговорень.
- Переглядати та очищувати завершені пункти кожен спринт.

## Бэклог кандидатів у ADR

1. Конверт повідомлення та політика версіонування схем.
2. Стандарт топології RabbitMQ.
3. Політика життєвого циклу модулів та сумісності.
4. Семантика обробки помилок та повторних спроб.
5. Базова лінія спостережності (журнали, метрики, трейси).
6. Межа криптографії payload-ів імплантів та політика управління ключами.
