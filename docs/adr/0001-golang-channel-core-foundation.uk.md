# ADR-0001: Екосистема базових пакетів Golang Channel Core

## Статус
Accepted

## Контекст

Vibe C2 орієнтований на спільноту і має на меті дозволити будь-кому створювати модулі у стилі vibe-code та розгортати інфраструктуру у стилі vibe-deploy.

Без багаторазового базового SDK кожна нова реалізація каналу повинна заново створювати:
- обробку канонічних повідомлень
- парсинг/зіставлення профілів обфускації
- валідацію профілів та перевірку конфліктів
- поведінку sync-клієнта (таймаути/повторні спроби)
- інтерфейси управління

Це уповільнює учасників і створює неузгоджену поведінку між каналами.

Система також має специфічні обмеження:
- канальні модулі повинні залишатися сліпими до відкритого тексту
- комунікація між імплантом/сесією та C2 зашифрована на рівні додатка
- площина даних канал<->C2 використовує HTTP sync (`POST /api/channel/sync`)
- управління профілями обфускації залишається під контролем каналу і надається C2 через RabbitMQ RPC

## Рішення

Прийняти екосистему пакетів з чітким розділенням відповідальностей:

1. `vibe-c2-golang-protocol` (спільні контракти)
   - канонічні структури повідомлень (`inbound.agent_message`, `outbound.agent_message`)
   - допоміжні функції версіонування та валідації
   - спільні коди статусів/помилок

2. `vibe-c2-golang-channel-core` (SDK середовища виконання каналів)
   - конвеєр виконання: transport -> de-obfuscate -> canonicalize -> sync -> re-obfuscate -> transport
   - механізм профілів (парсинг YAML, семантична валідація, виявлення перетинів)
   - стратегія вибору профілю (спочатку підказка `profile_id`, оптимізація кешу/частоти, потім перебір увімкнених профілів)
   - sync-клієнт для `POST /api/channel/sync`
   - поверхня управління RabbitMQ RPC для операцій з профілями
   - хуки телеметрії та типізована модель помилок

3. Опціональний майбутній пакет: `vibe-c2-golang-channel-adapters`
   - багаторазові транспортні адаптери для поширених каналів

### Основні правила реалізації

- Канали ніколи не розшифровують `encrypted_data`.
- Канонічний контракт channel-core залишається `id` + `encrypted_data`.
- Обфускація — це формування транспорту, а не криптографічна заміна.
- Вибір профілю не повинен залежати від профілю за замовчуванням; використовуйте спочатку підказку, потім перебір увімкнених профілів.
- Операції створення/оновлення профілів повинні відхиляти неоднозначні перетини.

## Наслідки

### Позитивні

- Значне зниження бар'єру для учасників спільноти.
- Узгоджена поведінка та семантика помилок між канальними модулями.
- Багаторазові, тестовані компоненти зменшують дублювання логіки.
- Легше зміцнення безпеки та спостережуваність по всіх каналах.

### Негативні / Витрати

- Початкові витрати на проєктування та підтримку пакетів.
- Необхідність підтримувати сумісність версій protocol та channel-core.
- Потрібні чіткі настанови для учасників та дисципліна тестування.

### Компроміси

- Більше структури проти швидшого спонтанного кодування.
- Початкова складність забезпечує довгострокову масштабованість та швидкість роботи учасників.

## Примітки щодо реалізації

Основні фази впровадження:
1. Контракти пакета Protocol та базова валідація.
2. Інтерфейси та каркас channel-core.
3. Механізм профілів + зіставлення + стратегія перебору.
4. Sync-клієнт + управлінський RPC.
5. Зміцнення безпеки, бенчмарки та приклади.
