# Чернетка архітектури

## Компоненти верхнього рівня

- **Core Server**: центральна площина управління для автентифікації, оркестрації, застосування політик та життєвого циклу модулів.
- **Operator UI / API Client**: інтерфейс для оператора для призначення задач, моніторингу та адміністрування.
- **Module Services**: незалежні сервіси, що реалізують можливості (наприклад: диспетчер задач, обробник результатів, сервіс сповіщень).
- **Message Bus (RabbitMQ)**: асинхронний backbone комунікації між основним сервером та модулями через канали/черги.
- **Persistence Layer**: довготривале сховище для користувачів, агентів, задач, стану модулів та журналів аудиту.
- **Telemetry / Logging**: централізований збір операційних подій та подій безпеки.

## Модель володіння комунікацією

- Логічна розмова — це **імплант/сесія ↔ ядро C2**.
- Модуль каналу є лише транспортним ретранслятором між імплантом/сесією та ядром C2.
- Модулі каналів переміщують непрозорі зашифровані блоби та мінімальні метадані маршрутизації (`id`).
- Модулі каналів не є протокольними учасниками бізнес-логіки імпланта та повинні залишатися "сліпими" до відкритого тексту.

## Модель комунікації

- Модулі за замовчуванням не викликають один одного напряму.
- Основний сервер та модулі взаємодіють через канали RabbitMQ.
- Payload-и імплант-до-C2 шифруються наскрізно на рівні застосунку.
- Модулі каналів не розшифровують payload-и імплантів; вони ретранслюють метадані + зашифровані блоби.
- Ядро надає виділений HTTP-ендпоінт синхронізації каналу (`POST /api/channel/sync`).
- Трафік імплантів канал->ядро працює за моделлю запит/відповідь через HTTP.
- При вхідному запиті ядро повертає зашифрований вихідний payload у тій самій HTTP-відповіді (тасків може бути вбудовано у шифротекст).
- Канал застосовує налаштовувані профілі обфускації для маппінгу транспортних полів <-> канонічних полів (`id`, `encrypted_data`).
- Управління профілями між ядром та каналом використовує RabbitMQ RPC керуючі виклики.
- Лише основні сервіси C2 мають ключі розшифрування та виконують операції розшифрування/перевірки.
- Повідомлення повинні мати версіоновану схему та бути ідемпотентними, де це можливо.
- Критичні потоки повинні використовувати підтвердження/повторні спроби та dead-letter черги.

## Початкова форма розгортання

- Docker Compose оркеструє всі сервіси для MVP-середовищ.
- Сервіси включають: основний сервер, RabbitMQ, базу даних та обрані модулі.
- Опціональний зворотний проксі може обробляти TLS-термінацію для ендпоінтів оператора/API.

## Межі довіри

- Мережа Інтернету/агентів до точки входу основного сервера.
- Мережа оператора до точки входу основного сервера.
- Основний сервер/модулі до RabbitMQ.
- Основний сервер/модулі до бази даних.

## Базові засоби безпеки

- Шифрований транспорт для всього зовнішнього трафіку/трафіку площини управління.
- Наскрізне шифрування на рівні застосунку для payload-ів імплантів (`implant` ↔ `core C2`).
- Ключі розшифрування доступні лише основним сервісам C2.
- Модулі каналів працюють як "сліпі" маршрутизатори для зашифрованих блобів.
- RBAC для всіх дій оператора.
- Контроль доступу до черг для кожного сервісу/модуля в RabbitMQ.
- Суворе журналювання аудиту для чутливих робочих процесів та дій, ініційованих модулями.

## Відкладені рішення

1. Конвенції найменування exchange/queue у RabbitMQ та стратегія routing-key.
2. Формат контракту повідомлень та політика версіонування.
3. Двигун бази даних та стратегія розділення схеми.
4. Політика пакування модулів та їхнього життєвого циклу (увімкнення/вимкнення/сумісність версій).
