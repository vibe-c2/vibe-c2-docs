# Типи модулів

Ця сторінка визначає початкові категорії модулів у модульній архітектурі Vibe C2.

## 1) Модулі каналів

Модулі каналів забезпечують транспортні шляхи між імплантами/сесіями та основною платформою C2.

### Обов'язки

- Приймати вхідний трафік агентів з конкретного транспорту/платформи.
- Витягувати/підтримувати мінімальні метадані маршрутизації (`id` + контекст каналу).
- Обробляти payload імпланта як непрозорий зашифрований блоб (без розшифрування/інспекції).
- Надсилати вхідний HTTP-запит до ендпоінту синхронізації ядра (`POST /api/channel/sync`).
- Очікувати HTTP-відповідь від ядра та ретранслювати повернений зашифрований payload назад імпланту/сесії.
- Реалізовувати транспортно-специфічну доставку відповідей (`poll`/long-poll/webhook reply/тощо).
- Завантажувати та застосовувати профілі обфускації (витягувати/повторно вбудовувати `id` та `encrypted_data`).
- Визначати профіль за транспортною підказкою `profile_id`, коли вона присутня; інакше зіставляти з увімкненими профілями.
- Зберігати та керувати YAML-профілями обфускації.
- Надавати RabbitMQ RPC дії управління для CRUD/активації/валідації профілів.
- Підтримувати статистику використання та кеш прив'язки джерел для оптимізації вибору профілів.
- Обробляти транспортно-специфічні проблеми (сесії, інтервали опитування, повторні спроби, обмеження частоти).

### Приклади

- HTTP(S) канал
- Telegram канал
- GitHub канал
- DNS канал (майбутнє)
- WebSocket канал (майбутнє)

### Примітки

- Тримати транспортну логіку ізольованою від бізнес-логіки/логіки задач.
- Модулі каналів за задумом є "сліпими" до відкритого тексту імплантів.
- Роль каналу — переміщення пакетів/блобів + надійність HTTP-ретрансляції, а не семантика C2.
- Справжній протокольний учасник — це ядро C2, а не модуль каналу.
- Забезпечувати автентифікацію та захист від зловживань для кожного каналу.
- Надавати метрики здоров'я каналу та затримки черги.

## 2) Модулі постачальників імплантів

Модулі постачальників імплантів визначають сімейства імплантів та поведінку їхнього життєвого циклу.

### Обов'язки

- Збирати/генерувати артефакти імплантів для цільової платформи/профілю.
- Визначати набір команд, що підтримується конкретним сімейством імплантів.
- Попередньо обробляти команди оператора/ядра у формат, специфічний для імпланта.
- Постобробляти необроблені відповіді імплантів у структуровані події/результати.
- Керувати метаданими/можливостями імплантів та версіями сумісності.

### Приклади

- Go постачальник імплантів
- .NET постачальник імплантів
- Python постачальник імплантів

### Примітки

- Тримати граматику команд, специфічну для постачальника, інкапсульованою.
- Версіонувати контракти команд для кожного постачальника імплантів.
- Відстежувати прапорці можливостей для кожної збірки імпланта (підтримувані команди/функції).

## 3) Модулі-перекладачі

Модулі-перекладачі перекладають між "мовою імпланта" та внутрішньою мовою/моделлю C2.

### Обов'язки

- Перекладати загальні наміри команд C2 у форми команд, специфічних для імпланта.
- Перекладати відповіді, специфічні для імпланта, у нормалізовану схему результатів C2.
- Обробляти двонаправлений маппінг, нормалізацію полів та переклад кодів помилок.
- Підтримувати кілька версій постачальників через явні профілі маппінгу.

### Приклади

- Загальна модель задач -> карта команд Go implant v1
- Відповіді Go implant v1 -> нормалізована модель результатів C2

### Примітки

- Перекладачі повинні бути детерміністичними та мати велике покриття тестами.
- Уникати змішування транспортних обов'язків у модулях-перекладачах.
- Тримати маппінги явними; уникати прихованої евристичної поведінки на критичних шляхах.

## Наскрізні вимоги до модулів

Усі типи модулів повинні:

- Взаємодіяти через канали RabbitMQ, використовуючи версіоновані схеми повідомлень.
- Включати ідентифікатори кореляції для відстежуваності між сервісами.
- Бути незалежно розгортаними/замінними у стеках Docker Compose.
- Генерувати структуровані журнали та сигнали здоров'я.
- Підтримувати graceful shutdown та ідемпотентну обробку, де це можливо.

Правила криптографічної межі:

- Криптографія payload-ів імплантів (`encrypt/decrypt/verify/sign`) належить основним сервісам C2.
- Модулі каналів повинні бути "сліпими" до відкритого тексту та обробляти лише метадані маршрутизації + зашифровані блоби.

## Рекомендований наступний крок

Створити ADR, що фіксує:

1. Базовий формат конверта повідомлення (`message_id`, `correlation_id`, `type`, `version`, `payload`, `timestamp`).
2. Конвенцію найменування exchange/queue.
3. Межі володіння між модулями канал/постачальник/перекладач.
