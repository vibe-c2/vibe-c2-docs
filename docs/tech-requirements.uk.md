# Технічні вимоги

## Функціональні вимоги

## FR-01: Автентифікація оператора

- Система повинна підтримувати вхід оператора із безпечною обробкою облікових даних.
- Система повинна підтримувати рольову авторизацію для привілейованих операцій.

## FR-02: Реєстрація агентів

- Система повинна підтримувати контрольоване зарахування агентів.
- Система повинна зберігати метадані ідентичності агентів та статус їхнього стану.

## FR-03: Призначення задач та потік результатів

- Оператори повинні мати можливість створювати та ставити задачі в чергу.
- Агенти повинні опитувати, виконувати задачі та надійно повертати результати.
- Система повинна зберігати історію задач із мітками часу.

## FR-04: Модульна сервісна архітектура

- Система повинна надавати **основний сервер** як центральний компонент оркестрації/управління.
- Система повинна підтримувати незалежні **модулі** для обробки, специфічної для конкретних можливостей.
- Модулі повинні взаємодіяти через спільну шину повідомлень та залишатися незалежно розгортаними.

## FR-05: Backbone обміну повідомленнями на RabbitMQ

- Основний сервер та модулі повинні взаємодіяти через канали/черги RabbitMQ.
- Трафік імплантів між каналом і ядром повинен використовувати запит/відповідь через HTTP (`POST /api/channel/sync`).
- HTTP-відповідь ядра повертає зашифрований вихідний payload для того самого `id` (або зашифрований no-op payload).
- Топологія черг повинна підтримувати маршрутизацію за типом повідомлення та зоною відповідальності модуля.
- Критичні потоки повідомлень повинні підтримувати підтвердження та поведінку повторних спроб/dead-letter.

## FR-06: Шифрування імплантів на рівні застосунку

- Payload-и імплант/сесія-до-C2 повинні бути зашифровані на рівні застосунку наскрізно.
- Модулі каналів не повинні розшифровувати payload-и імплантів.
- Модулі каналів можуть обробляти лише мінімальні метадані маршрутизації (`id`, метадані каналу) та зашифровані блоби.
- Модулі каналів діють як транспортні ретранслятори та не повинні володіти семантикою протоколу C2.
- Основні сервіси C2 повинні отримувати ключовий матеріал/контекст за допомогою `id` та виконувати розшифрування/перевірку.

## FR-07: Профілі обфускації каналів

- Модулі каналів повинні підтримувати кілька увімкнених профілів обфускації на канал.
- Профілі повинні визначати правила маппінгу/обфускації для `profile_id` (необов'язкова підказка), `id` та `encrypted_data` у транспортних полях.
- Профілі повинні зберігатися у сховищі як YAML-документи.
- Модулі каналів повинні надавати RabbitMQ RPC дії управління для CRUD/активації/валідації профілів.
- Канал повинен перебирати всі увімкнені профілі методом brute-force, коли підказка відсутня або недійсна.

## FR-08: Вирішення збігу обфускації та продуктивність

- На вході канал повинен спочатку визначити профіль за транспортною підказкою `profile_id`, коли вона доступна.
- Якщо підказка відсутня/недійсна, канал повинен вибрати профіль шляхом зіставлення увімкнених профілів та декодування до канонічних полів.
- Створення/оновлення профілю повинно валідувати та відхиляти неоднозначне перекриття з існуючими увімкненими профілями.
- Канал повинен оптимізувати вибір профілю, використовуючи частоту використання та кеш прив'язки джерела до профілю.

## FR-09: Можливість аудиту

- Дії, що стосуються безпеки, повинні журналюватися із зазначенням актора, дії та часу.
- Журнали повинні бути доступні для запитів при розслідуванні інцидентів.
- Дії, ініційовані модулями, повинні бути відстежуваними за ідентифікаторами повідомлень/кореляції.
- Зміни профілів обфускації повинні піддаватися аудиту.

## Нефункціональні вимоги

## NFR-01: Безпека

- Увесь трафік площини управління повинен використовувати шифрування при передачі.
- Конфіденційність/цілісність payload-ів імплантів повинна забезпечуватися шифруванням на рівні застосунку.
- Ключі розшифрування повинні бути ізольовані в основних сервісах C2 (не в модулях каналів).
- Секрети не повинні зберігатися у відкритому вигляді.
- Події автентифікації повинні піддаватися аудиту.

## NFR-02: Надійність

- Цільовий аптайм основного сервера: >= 99,5% для MVP-середовищ.
- Стан задач та стан обміну повідомленнями повинні переживати перезапуски процесів.
- Збої модулів повинні бути ізольованими та не повинні призводити до краху непов'язаних сервісів.

## NFR-03: Продуктивність

- Цільовий час відповіді на команду оператора: p95 < 500 мс для операцій з метаданими.
- Обробка check-in агентів повинна масштабуватися до очікуваного навантаження MVP.

## NFR-04: Підтримуваність

- Кодова база повинна включати автоматизацію лінтингу/тестування.
- Ключові проєктні рішення повинні бути задокументовані через ADR.

## Обмеження

- Документація та CI/CD розміщені на GitHub.
- Архітектура середовища виконання контейнеризована та оркестрована за допомогою Docker Compose для MVP.
- Обмін повідомленнями між основним сервером та модулями використовує RabbitMQ.
- Вибір технологій повинен пріоритизувати швидку ітерацію для MVP.
