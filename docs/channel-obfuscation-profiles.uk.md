# Профілі обфускації каналу

Ця сторінка визначає шар обфускації, який використовують модулі каналів при транспортуванні `id` (вхідні) та `encrypted_data` (в обох напрямках) для комунікації імпланту/сесії з ядром C2.

## Призначення

Профілі обфускації — це шар формування транспорту поверх базового контракту синхронізації. Вони **не** замінюють шифрування на рівні додатку між імплантом і C2.

- Шифрування на рівні додатку (`encrypted_data`) залишається обов'язковим.
- Обфускація контролює, як `id` та `encrypted_data` вбудовуються в транспортні поля каналу.

## Що може контролювати профіль

Профіль може визначати, де і як розміщуються поля, наприклад:

- HTTP-заголовки
- параметри запиту
- cookies
- поля тіла
- змішане розміщення (розподіл між кількома місцями)

Окрім `id` та `encrypted_data`, транспорт може нести підказку `profile_id`, що використовується для швидкого вибору профілю.
Профіль також може визначати оборотне кодування/обгортання, наприклад:

- варіанти base64/base64url
- перейменування/створення псевдонімів полів
- опціональне обгортання/шифрування на рівні каналу

Профіль також визначає поведінку каналу через об'єкт `action` — `action.type` вибирає поведінку після зіставлення профілю та канонічного декодування, `action.params` несе визначені каналом параметри. Див. [довідник YAML](channel-obfuscation-yaml-reference.md) для точної структури.

> **Статус:** специфікація на рівні документації (цільовий дизайн). Реалізація в рантаймі може відставати від специфікації.

## Зберігання та власність профілів

- Профілі зберігаються у довговічному сховищі.
- Формат профілів — YAML.
- Модуль каналу відповідає за завантаження, валідацію та виконання профілів у рантаймі.
- Ядро C2 відповідає за UX/API управління та рішення щодо політик.

Використовуйте сторінку прикладів для самодостатнього покрокового огляду та каталогу готових профілів.

## Потік у рантаймі

Вхідний (імплант/сесія -> канал -> ядро):

1. Канал отримує обфусковане транспортне корисне навантаження.
2. Канал спочатку намагається витягти підказку `profile_id` з налаштованих місць розташування підказок.
3. Якщо підказка валідна і вказує на увімкнений профіль, канал використовує цей профіль напряму.
4. Якщо підказка відсутня/невалідна, канал формує список кандидатів із увімкнених профілів.
5. Канал перебирає кандидатів, впорядкованих за частотою зіставлень у рантаймі.
6. При першому успішному декодуванні до канонічних полів (`id`, `encrypted_data`) канал використовує цей профіль.
7. Якщо жоден кандидат не підійшов, канал відхиляє запит як незіставлений профіль.
8. Після зіставлення профілю канал визначає та виконує `action` з цього профілю.
9. Для дій типу process канал надсилає канонічний запит до кінцевої точки синхронізації C2 після успішного визначення дії.

Вихідний (ядро -> канал -> імплант/сесія):

1. Канал отримує канонічну відповідь (`outbound.agent_message`), що містить лише `encrypted_data`.
2. Канал використовує профіль, визначений на вхідному етапі, та зіставлення `encrypted_data_out` для формування вихідного транспортного корисного навантаження. Вихідні дані не несуть `id`.
3. Канал повертає відповідь у транспортній формі до імпланту/сесії.
4. Канал оновлює лічильники використання профілю/кеш для майбутнього впорядкування.

## Приклади дій

Стандартні дії (кожен канал повинен реалізувати):

- `sync` — декодувати зіставлення профілю, викликати кінцеву точку синхронізації C2, закодувати відповідь.

Користувацькі дії (визначені каналом, приклади):

- `redirect` (HTTP) — повертає поведінку перенаправлення на альтернативну інфраструктуру каналу замість локальної обробки синхронізації.

Див. [довідник YAML профілів обфускації](channel-obfuscation-yaml-reference.md) для повних прикладів профілів з конфігурацією дій.

## Модель управління

Модулі каналів надають кінцеві точки управління через RabbitMQ RPC, щоб ядро могло керувати профілями.

### Приклади RPC-дій

- `obf.profile.list`
- `obf.profile.get`
- `obf.profile.create`
- `obf.profile.update`
- `obf.profile.delete`
- `obf.profile.activate`
- `obf.profile.validate`
- `obf.profile.simulate_match`
- `obf.profile.stats`

Валідація при створенні/оновленні повинна включати перевірку перекриттів з іншими увімкненими профілями для того самого каналу.

## Стратегія конфліктів зіставлення та продуктивності

### Контроль конфліктів

- Кілька увімкнених профілів можуть випадково зіставитися з однаковою вхідною формою.
- Канал/C2 повинні відхиляти операції створення/оновлення, що вносять неоднозначне перекриття (для того самого каналу).
- Підказка `profile_id` повинна однозначно відповідати одному увімкненому профілю в межах каналу.
- Поведінка профілю за замовчуванням не підтримується; незіставлені корисні навантаження повинні бути відхилені.

### Контроль продуктивності

- Надавати перевагу прямому вибору через транспортну підказку `profile_id`.
- Впорядковувати спроби за спостережуваною частотою/рівнем успіху.
- Кешувати спорідненість джерела до профілю (наприклад, `source_ip -> profile_id`, `telegram_chat_id -> profile_id`) з TTL.
- При влучанні в кеш спочатку пробувати кешований профіль; при промаху/невдачі продовжувати впорядкованими кандидатами.

## Межі безпеки

- Обфускація не є заміною криптографії.
- Канал повинен залишатися сліпим до відкритого тексту бізнес-семантики імпланту.
- Логіка профілю не повинна вимагати розшифрування `encrypted_data`.
- Зміни активного профілю повинні бути аудитовані (хто/коли/що змінив).
