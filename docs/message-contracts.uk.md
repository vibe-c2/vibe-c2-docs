# Контракти повідомлень

Ця сторінка визначає HTTP-контракти синхронізації для комунікації імпланту/сесії між каналом та ядром C2.

## Модель

- Канонічна модель каналу-ядра складається лише з `id` + `encrypted_data`.
- Транспорт імпланту/каналу може використовувати обфусковане розміщення (заголовки/параметри запиту/cookies/тіло/тощо), визначене активним профілем.
- Канал нормалізує транспортне корисне навантаження до канонічної форми перед викликом ядра.
- Канал надсилає HTTP-запит до ядра: `POST /api/channel/sync`.
- Ядро відповідає повідомленням `outbound.agent_message`, що містить зашифровані дані.
- Канал застосовує зіставлення профілю для вихідної транспортної відповіді.

---

## `inbound.agent_message` (тіло HTTP-запиту)

Призначення: запит від каналу до ядра, що несе вхідні зашифровані дані імпланту.

### Тіло (v1)

```json
{
  "message_id": "01JNX6R8VQ2H3CN4K9EJ1T2Z7M",
  "type": "inbound.agent_message",
  "version": "1.0",
  "timestamp": "2026-03-09T21:05:12.481Z",
  "source": {
    "module": "channel-http",
    "module_instance": "channel-http-1",
    "transport": "http",
    "tenant": "default"
  },
  "id": "s-2b77df",
  "encrypted_data": "QkM4V1R...",
  "meta": {
    "receive_count": 1,
    "trace_id": "tr-6fd92d8b"
  }
}
```

### Правила валідації

На стороні каналу (імплант -> канал):
- Вимагати непорожній `id`.
- Вимагати непорожній `encrypted_data`.
- Трактувати `encrypted_data` як непрозорі дані.

На стороні ядра (тіло HTTP-запиту):
- Вимагати `type == inbound.agent_message`.
- Вимагати `id` та `encrypted_data`.
- Вимагати `timestamp` у форматі RFC3339, що піддається парсингу.

---

## `outbound.agent_message` (тіло HTTP-відповіді)

Призначення: відповідь від ядра до каналу для того самого `id`, що несе лише зашифровані вихідні дані.

### Тіло (v1)

```json
{
  "message_id": "01JNX7D8H8QY3G6P2R4X1K8ABC",
  "type": "outbound.agent_message",
  "version": "1.0",
  "timestamp": "2026-03-09T21:05:12.690Z",
  "id": "s-2b77df",
  "encrypted_data": "U2FtcGxlRW5jcnlwdGVkQmxvYg==",
  "meta": {
    "status": "ok",
    "trace_id": "tr-6fd92d8b"
  }
}
```

### Семантика відповіді

- Відповідь містить лише одне поле корисного навантаження: `encrypted_data`.
- `encrypted_data` є непрозорим для каналу.
- Коли немає очікуваної роботи, ядро повертає порожнє/no-op зашифроване корисне навантаження.
- Канал передає `encrypted_data` до імпланту/сесії без змін.

### Правила валідації

На стороні каналу (тіло HTTP-відповіді):
- Вимагати `type == outbound.agent_message`.
- Вимагати відповідний `id`.
- Вимагати поле `encrypted_data`.
- Трактувати `encrypted_data` як непрозорі дані.

---

## Очікування щодо обробки

- Канал надсилає `inbound.agent_message` до `POST /api/channel/sync` для кожного вхідного повідомлення імпланту/сесії.
- Ядро обробляє вхідне корисне навантаження та повертає `outbound.agent_message` у тому ж HTTP-обміні.
- Ядро розшифровує/верифікує вхідні дані та шифрує вихідні; канал ніколи не розшифровує.
- Ядро може повернути порожнє/no-op зашифроване корисне навантаження, коли для даного `id` немає очікуваних завдань.
